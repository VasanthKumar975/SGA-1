Question 4: File Access and I/O Investigation

Commands and outputs:

1) rm -f app.log stdout.txt stderr.txt combined.txt
2) echo "sample log line" > app.log
3) lsof | head -n 20
Output excerpt:
COMMAND PID USER FD TYPE DEVICE SIZE/OFF NODE NAME
...

4) exec 3<app.log
5) lsof -p $$
Output excerpt:
bash 13434 codespace 3r REG 7,4 16 1310807 /workspaces/SGA-1/q4/app.log

6) ls -l /proc/$$/fd | grep app.log
Output:
lr-x------ 1 codespace codespace 64 Jul  3 16:31 3 -> /workspaces/SGA-1/q4/app.log

7) ls -l /proc/$$/fd
Output:
0 -> /dev/pts/3
1 -> /dev/pts/3
2 -> /dev/pts/3
3 -> /workspaces/SGA-1/q4/app.log
6 -> /dev/null
29 -> /home/codespace/.vscode-remote/data/logs/20260703T161230/mcpGateway.log
255 -> /dev/pts/3

8) ls -l > stdout.txt
9) ls /not_a_real_path 2> stderr.txt
10) (ls -l && ls /not_a_real_path) > combined.txt 2>&1
11) ulimit -a
Output excerpt:
open files (-n) 524288
stack size (kbytes, -s) 8192
file size (blocks, -f) unlimited

12) exec 3<&-

Observations:
The shell opened a file descriptor for app.log and exposed it through /proc/$$/fd. Standard output and error were redirected into files, and the combined redirection demonstrated how both stdout and stderr can be captured together. The resource limits output showed that the shell environment allows a large number of open files and has no practical file-size restriction.
